# OneThought（Tauri 版）开发问题与经验记录

本文档整理本软件在设计与实现过程中遇到的主要问题、原因与解决思路，便于日后回顾与团队学习。

---

## 1. 构建与开发环境

| 问题 | 原因 | 解决 |
|------|------|------|
| `npm run tauri build` 报 `--ci` 参数错误 | CI 环境下 Tauri 对 `ci` 标志的解析与本地脚本不一致 | 使用 `CI=true npm run tauri build` |
| 前端 TypeScript 报错：`path`、`__dirname` 等 | Vite 配置在 `tsconfig.node` 中未包含 Node 类型 | 安装 `@types/node`，在 `tsconfig.node.json` 的 `compilerOptions.types` 中加入 `"node"` |
| 端口被占用 | 本地 dev 服务器端口冲突 | 结束占用进程或更换 Vite 端口 |

**经验**：Tauri 项目建议区分 `dev` 与 `release` 脚本，并在文档中写明正式打包命令。

---

## 2. 权限与能力（Capabilities）

| 问题 | 原因 | 解决 |
|------|------|------|
| 快捷记录弹窗无法 `hide()`、按钮/Esc 无效 | 子窗口未授予 `core:window:allow-hide` 等权限 | 在 `capabilities/quick-capture.json` 中为 `quick_capture` 窗口声明所需权限 |
| 拖拽窗口无效 | 缺少 `core:window:allow-start-dragging` | 同上，按需补充权限 |

**经验**：Tauri 2 按窗口拆分能力，新窗口要在独立 capability 里列全会用到的 API。

---

## 3. 全局快捷键与多窗口

| 问题 | 原因 | 解决 |
|------|------|------|
| 热键会干扰主窗口显示/隐藏 | 热键逻辑直接操作主窗口 | 单独建 `quick_capture` 窗口，热键只 `show`/`focus` 该窗口 |
| Windows 上快捷键冲突导致进程崩溃 | `on_shortcut` 失败时用 `?` 向上传播错误，setup 失败即 panic | 对注册结果 `match`，失败只打日志并提示用户在设置里改键 |
| 多开 portable 互相干扰 | 无单实例约束 | 使用 `tauri-plugin-single-instance`，二次启动时聚焦已有实例主窗口 |

**经验**：全局快捷键在 Windows 上极易与浏览器、IDE 冲突，必须**失败可恢复**，不能作为启动必要条件。

---

## 4. 快捷记录弹窗 UI（跨平台）

| 问题 | 原因 | 解决 |
|------|------|------|
| 多余白边/透明区域 | 窗口尺寸大于内容、`#root` 100vh、卡片 margin | `ResizeObserver` + `setSize`/`center` 贴合内容；CSS 让卡片铺满宽度 |
| macOS 黑边、圆角外漏 | 系统窗口阴影与 HTML 圆角不一致 | 曾尝试 `shadow: false`、透明窗口；最终为稳定性采用不透明卡片方案 |
| Windows 上 Esc 不关闭 | 只在 `textarea` 上监听 | 在 `window` 上增加全局 `keydown` 监听 Escape |
| 行内快捷键说明 | 文档与默认值 | 默认热键改为 `Alt+T`，减少与 `Ctrl+T` 冲突 |

## 5. Windows 便携版与 WebView2

| 问题 | 原因 | 解决 |
|------|------|------|
| 公司机无 Edge、无法联网装 Runtime | 依赖系统 WebView2 | 使用 **WebView2 Fixed Version**，与 exe 同目录放置，启动前设置 `WEBVIEW2_BROWSER_EXECUTABLE_FOLDER` |
| 启动即闪退、难以排查 | 无控制台、无日志 | Rust 侧写 `onethought.log`（exe 同目录）、`panic::set_hook`、关键步骤 `app_log` |
| 首次运行只有托盘、主界面不出现 | WebView2 首次初始化慢；`CloseRequested` 被拦截为隐藏且无日志 | 主窗口显式 `visible: true`；setup 结束对主窗口 `show()` + `set_focus()`；主窗口事件打日志（含 `CloseRequested`） |
| 安装到 `Program Files` 后弹窗「无法创建数据目录」、路径指向 `...\WebView2Data\EBWebView` | 曾将 `WEBVIEW2_USER_DATA_FOLDER` 固定为 exe 同目录；**Program Files 普通用户无写权限** | **仅在使用 Fixed Version 时**尝试 `exe 旁/WebView2Data`；先检测目录可写，失败则回退到 `%LOCALAPPDATA%\com.onethought.tauri\WebView2Data`；**仅用系统 WebView2 时不设置**该变量，交给 Edge 默认用户目录 |

**经验**：`WEBVIEW2_USER_DATA_FOLDER` 不能默认指向 `Program Files` 下路径。Fixed Version 必须把 `WEBVIEW2_BROWSER_EXECUTABLE_FOLDER` 设在 **创建 WebView 之前**；用户数据目录：便携可写时用 exe 旁，否则用当前用户 LocalAppData。

## 6. 开机自启

| 问题 | 原因 | 解决 |
|------|------|------|
| 勾选「开机启动」重启后不生效 | 只把布尔值写进 `config.json`，未调用系统 API | 使用 `tauri-plugin-autostart`，保存配置时调用 enable/disable |
| 设置失败用户不知道 | 无错误反馈 | `invoke` 返回错误；前端展示红色提示并必要时还原勾选 |

---

## 7. 设置页「一直加载中」

| 问题 | 原因 | 解决 |
|------|------|------|
| 设置页永远显示「加载中…」 | `getConfig()` 失败时无 `.catch()`，`settingsForm` 永为 `null` | `.catch()` 设置错误状态；可选 5 秒超时提示查日志 |
| 后端无痕迹 | `config_get` 成功时无日志 | Rust `config_get` 打日志并返回 `Result`，便于区分 invoke 失败 |

## 8. AI / 大模型对接

| 问题 | 原因 | 解决 |
|------|------|------|
| 行内网关 URL、Body 与 OpenAI 不一致 | 行内为 `{base}{model}/v1/chat/completions`，`maxTokens` 为字符串等 | 增加 `llmMode`（行内/行外），分支构造 URL 与 JSON |
| 日志里按字节截断导致 panic | `&s[..500]` 截在 UTF-8 多字节字符中间 | `truncate_chars` 按字符数安全截断 |
| 行内非流式改流式 | 网关支持 `stream: true` 与 SSE | 行内请求体增加 `stream: true`，与行外共用 SSE 解析逻辑 |
| AI 总结未按界面筛选 | 用了全量 `allThoughtsForTags` | 改为 `groups.flatMap(g => g.items)`，与标签+时间一致 |
| 分类/描述/方案挤成一行 | `ReactMarkdown` 把单个 `\n` 当空格 | AI 总结区改用 `<pre className="ai-summary-pre">` + `white-space: pre-wrap` |

## 9. 前端状态与导航

| 问题 | 原因 | 解决 |
|------|------|------|
| 偶尔进入设置页空白 | 先切到设置，`config` 尚未加载完成，`settingsForm` 未赋值 | `useEffect` 在 `view === "settings" && config` 时补全 `settingsForm` |

## 10. 通用工程习惯（本仓库实践）

1. **错误要可见**：Promise 必须有 `catch` 或等价处理，避免「静默失败 + 无限 loading」。
2. **字符串截断**：任何面向用户的日志/错误信息，避免对 `String`/`str` 按字节索引截断中文。
3. **跨平台**：窗口透明、阴影、快捷键行为在 Windows / macOS 上差异大，需分别验证或选择保守方案。
4. **可观测性**：关键路径（启动、WebView2、配置读写、窗口显示/隐藏、AI 请求）写文件日志，便于无控制台环境排错。

## 11. 版本与发布

- 使用 GitHub Actions 在 `windows-latest` / `macos-latest` 上构建，通过 **tag**（如 `v0.3.x`）触发 Release。
- Windows 同时提供安装包与 **portable** 可复制 exe，便于受限环境试用。
- GitHub 对基于 Node 20 的 Action 运行时有弃用提示时：可在 workflow 顶层设置 `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: "true"`，并将 `setup-node` 的 `node-version` 提升到 22 LTS，减少 Annotations 警告。
